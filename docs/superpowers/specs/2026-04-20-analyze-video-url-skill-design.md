# 视频链接分析 Skill 设计文档

**日期**：2026-04-20
**状态**：修订中（根据 Codex Review 反馈更新）

---

## 概述

创建一个 Claude Code Skill（`analyze-video-url`），当用户提供视频分享链接或播放页面 URL 时，自动分析渠道归属，抓取页面内容，判断是已有渠道的新链接类型还是需要新增解析渠道，并生成对应的解析器代码模板。

---

## 元信息与触发条件

### Skill 信息

- **名称**：`analyze-video-url`
- **位置**：Claude Code Skill 文件（配置在 skills 目录中）

### 输入格式

同时支持两种输入：

1. **分享链接文本**：含 URL 的混合文本（如 `"复制的抖音分享文本 https://v.douyin.com/xxx"`）
2. **纯 URL**：完整的播放页面地址（如 `https://www.douyin.com/video/123456`）

### 渠道判断（三分类）

Skill 不只依赖原始 URL 的域名匹配，还会跟随重定向链、分析最终落地页结构来做判断：

1. **已知域名**：原始 URL 域名与 `parser/vars.go` 中 `videoSourceInfoMapping` 的 `VideoShareUrlDomain` 子串匹配成功 → 已有渠道，指明具体平台名
2. **未知域名 → 跳转命中**：原始域名不匹配，但跟随重定向后的落地页 URL 或页面结构命中已有平台（如某短链跳转到 `www.douyin.com`） → 已有渠道的新入口域名，需补充域名注册
3. **疑似新平台**：重定向链和落地页均不匹配任何已有平台 → 全新平台，需要新增解析器

**为什么需要三分类而不是二分类**：当前代码中多个平台的短链需要先跟随重定向才能识别真实目标。例如：
- 抖音 App 分享链接 `v.douyin.com` 重定向后可能跳到 `ixigua.com`（走西瓜解析），也可能跳到 `www.douyin.com`（`douyin.go:200-203`）
- Twitter 的 `t.co` 短链需要先跟随重定向获取真实 URL 后再抽 tweet ID（`twitter.go:21-35`）
- B 站的 `b23.tv` 短链同理（`bilibili.go:123-142`）
- 抖音 `parseShareUrl` 按 host 做精确分支，未覆盖的 host 直接报错（`douyin.go:162-169`）

仅用 `strings.Contains` 对原始字符串做域名匹配（`parser/parser.go:28`）会遗漏「已知平台但域名未注册」的场景，导致误判为新增平台并生成错误的新解析器。

## 分析流程

### 步骤 1：URL 提取

- 纯 URL → 直接使用
- 混合文本 → 参考 `utils/utils.go` 中 `RegexpMatchUrlFromString` 的正则逻辑提取 URL

### 步骤 2：渠道归属判断（多阶段）

#### 阶段 A：原始域名匹配

- 读取 `parser/vars.go` 中所有 `VideoShareUrlDomain`
- 对输入 URL 做子串匹配（与 `parser/parser.go:ParseVideoShareUrl` 一致）
- 匹配成功 → **已知域名**，进入步骤 3

#### 阶段 B：重定向链分析

- 原始域名不匹配时，用 HTTP 客户端（`NoRedirectPolicy`）跟随短链重定向
- 记录完整重定向链（每跳的 Location URL）
- 对重定向目标 URL 再次做域名匹配
- 匹配成功 → **未知域名 → 跳转命中**，记录需要补充的新域名

#### 阶段 C：页面结构分析

- 如果重定向链也不匹配，抓取最终落地页 HTML
- 检查页面中是否包含已有平台的数据特征（如 `window._ROUTER_DATA` 对应抖音、`window.INIT_STATE` 对应快手）
- 匹配成功 → **未知域名 → 跳转命中**，记录需要补充的新域名
- 均不匹配 → **疑似新平台**

### 步骤 3：页面抓取与结构深度分析

用 HTTP 客户端（模拟移动端 UA）请求目标 URL，分析：

- **重定向链**：完整记录每跳 URL，识别短链 → 中间跳 → 落地页的完整路径
- **重定向策略**：是否需要定制 redirect policy（如快手只在特定路径才跟随重定向，`kuaishou.go:22-29`）
- **Host 分发逻辑**：是否按 host 做分支处理（如抖音按 host 分 PC/App 分享链接，`douyin.go:162-169`）
- **嵌入 JSON**：HTML 中是否有 `window.__INITIAL_STATE__`、`window.INIT_STATE`、`window._ROUTER_DATA`、`$render_data` 等嵌入式数据？
- **API 模式**：是否有可提取的 API 端点？需要几次请求？（如 B 站需要先 view API 再 playurl API，`bilibili.go:24-44`）
- **URL 路径变换**：是否需要路径替换？（如快手 `/fw/long-video/` → `/fw/photo/`，`kuaishou.go:44-46`）
- **视频 ID 格式**：ID 在 URL 路径中的位置和格式，提取正则
- **关键字段**：视频 URL、封面图、作者信息等在页面/API 响应中的具体 JSON 路径
- **反爬特征**：是否需要特殊参数（如抖音图集的 `a_bogus`、`web_id`，`douyin.go:46-48`）

### 步骤 4：生成报告

在对话中输出结构化分析报告：

- **渠道判定**：三分类结果（已知域名 / 跳转命中已有渠道 / 疑似新平台）
- **URL 格式分析**：短链 → 重定向目标 → 最终 URL 结构
- **解析控制流**：重定向策略、host 分发、URL 变换等控制逻辑
- **数据提取策略**：推荐的解析模式（重定向提取 ID / HTML 嵌入 JSON / API 调用）
- **关键字段位置**：视频 URL、标题、封面、作者信息的具体路径（含 gjson 路径或 JSON 字段名）
- **HTTP 客户端选择**：推荐使用 `resty` 还是标准 `net/http`（参考同类平台的做法）

### 步骤 5：代码生成

- **已知域名（新链接类型）**：生成现有解析器需要扩展的部分（新 host 分支、新数据提取路径、新正则）
- **未知域名 → 跳转命中**：生成 `vars.go` 域名补充 + 可能的解析器 host 分支扩展
- **疑似新平台**：生成完整的解析器代码文件

### 步骤 6：代码审查与确认

**所有代码变更（包括 `vars.go`）都必须经用户确认后才写入文件**：

1. 在对话中展示完整的代码和变更说明（含所有涉及的文件）
2. 将所有变更组织为一次 patch，包括：
   - `parser/<platform>.go`（新解析器或已有解析器扩展）
   - `parser/vars.go`（常量和映射条目变更）
   - `parser/<platform>_test.go`（测试文件）
3. 用户确认后才执行文件写入
4. 不留"手动处理步骤"——所有变更都包含在 patch 中

## 代码模板规范

### 已有渠道扩展（已知域名 / 跳转命中）

生成的扩展代码必须覆盖完整的解析控制流，包含：

- `vars.go` 中 `VideoShareUrlDomain` 需新增的域名
- 解析器中新增的 host 分支处理逻辑（参考 `douyin.go:162-169` 的 switch-host 模式）
- 新的重定向策略（如需要定制 redirect policy，参考 `kuaishou.go:22-29`）
- 新的 URL 路径变换规则（如需要路径替换，参考 `kuaishou.go:44-46`）
- 新的数据提取路径（gjson 路径或 JSON 字段）
- **不得仅修改 `vars.go` 而忽略运行时的 host 分发和控制流逻辑**

### 新增渠道（完整解析器）

生成文件 `parser/<platform>.go`，根据分析结果选择最合适的实现模式：

**解析模式选择**：
- 短链重定向模式：参考抖音、Twitter（`NoRedirectPolicy` → 提取 Location → 提取 ID）
- HTML 嵌入 JSON 模式：参考快手（`window.INIT_STATE` → gjson 提取）
- API 调用模式：参考 B 站（构造 API URL → JSON 响应解析）
- 混合模式：如需要（参考抖音图集需要构造 API URL 并生成随机参数）

**代码规范**：
- 零值结构体（如 `type xxPlatform struct{}`）
- 实现 `videoShareUrlParser` 接口（`parseShareUrl` 方法）
- 可选实现 `videoIdParser` 接口（`parseVideoID` 方法）
- HTTP 客户端根据平台需要选择 `resty` 或标准 `net/http`（不要强制统一 resty）
- JSON 解析根据平台需要选择 `gjson` 或 `encoding/json`（不要强制统一 gjson）
- 移动端 User-Agent
- 中文注释标注关键流程

同时在 `vars.go` 中包含在 patch 里的变更：

- 新平台常量（如 `SourceXX = "xx"`）
- `videoSourceInfoMapping` 映射条目（含域名列表）

所有变更统一提交确认，不区分"自动"和"手动"。

### 测试文件

生成 `parser/<platform>_test.go`，包含：

- URL/ID 提取函数的纯单元测试（不依赖外部网络）
- 正则/路径匹配的单元测试
- **不默认生成真实 URL 的集成测试**，避免引入外部依赖导致测试波动（参考 `cmd/api_test.go:174` 的 `stubParserFuncs` 调用模式）
- 如果用户明确需要集成测试，使用 mock HTTP 响应而非真实 URL

## 成功标准

### 验证方式

Skill 的分析正确性通过 **人工抽样验证** 确认。以下 4 个场景定义了输入特征和验收条件，验证时由用户选择符合场景特征的真实 URL 逐项检查。场景本身是稳定的判定规则，具体样本随平台变化而更新。

**场景 1：已知域名，命中 `VideoShareUrlDomain`**
- **输入特征**：URL 域名直接匹配 `vars.go` 中已注册的域名（如 `www.bilibili.com`、`weibo.com`、`www.xiaohongshu.com`）
- **预期**：判定为「已知域名」，给出具体平台标识（如 `SourceBiliBili`）
- **验收**：报告中 `渠道分类` = 已知域名，`平台名` 为正确的 Source 常量

**场景 2：已知短链域名，重定向后平台以落地页为准**
- **输入特征**：URL 域名匹配已注册域名（如 `v.douyin.com`），但重定向落地页可能属于不同平台（如跳转到 `ixigua.com`，参考 `douyin.go:200-203`）
- **预期**：判定为「已知域名」，`平台名` 以重定向落地页域名为准（可能为抖音或西瓜）
- **验收**：报告中 `平台名` 必须与实际重定向目标匹配，而非固定写死原始域名对应的平台

**场景 3：未知域名，重定向后命中已有平台**
- **输入特征**：URL 域名不在任何 `VideoShareUrlDomain` 中，但跟随重定向后落地页域名匹配已知平台（如某新短链服务跳转到 `www.kuaishou.com`）
- **预期**：判定为「跳转命中」，给出目标平台标识，提示需在 `vars.go` 补充域名
- **验收**：报告中 `渠道分类` = 跳转命中，`平台名` 为落地页对应的平台，patch 中包含 `vars.go` 域名补充

**场景 4：未知域名，落地页也不匹配任何已有平台**
- **输入特征**：URL 域名和重定向落地页均不在已注册域名中，且页面结构不包含已知平台特征（如 `youtube.com`、`tiktok.com`）
- **预期**：判定为「疑似新平台」
- **验收**：报告中 `渠道分类` = 疑似新平台，生成完整的新解析器代码

**最低标准**：4 个场景全部正确分类。

### 报告必须输出的字段

每份分析报告必须包含以下字段，缺失任意一项视为不通过：

1. `渠道分类`：已知域名 / 跳转命中 / 疑似新平台（三选一）
2. `平台名`：已知域名和跳转命中时必须给出具体平台标识（如 `SourceDouYin`）
3. `重定向链`：完整的 URL 跳转路径（至少包含原始 URL 和最终落地页 URL）
4. `数据提取策略`：明确指出使用哪种解析模式（重定向提取 / HTML 嵌入 JSON / API 调用）
5. `关键字段路径`：视频 URL、标题、封面、作者信息在 JSON 中的 gjson 路径或字段名
6. `待修改文件清单`：所有需要变更的文件列表

### 代码质量标准

1. 新增渠道的代码模板必须 `go build` 编译通过
2. 已有渠道的扩展必须同时覆盖 `vars.go` 和对应解析器文件的变更
3. 生成的测试文件必须 `go test` 通过且不依赖外部网络
